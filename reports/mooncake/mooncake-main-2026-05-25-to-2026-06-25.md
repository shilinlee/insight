## 重点摘要

过去一个月 Mooncake 的主线变化不是简单的小修小补，而是围绕 **Store 多租户/SSD 分层存储成熟化**、**Transfer Engine/TENT 在更多硬件和网络环境下的可用性**、**异构平台扩展**、**部署与文档体系重构** 四条主线推进。

1. **Mooncake Store 从“对象缓存”继续向“可运营的多租户分层存储服务”演进**：新增租户配额核心、租户元数据隔离、租户身份贯穿对象 RPC、动态租户 quota 管理、quota master admission、admin/eviction/metrics/snapshot 补齐，以及 SSD offload 的部署、分配、回收和指标修复。
2. **SSD / NoF / 分层缓存能力明显加强**：Store 增加 SSD free-ratio-first 分配策略、standalone SSD offload 配置、NoF SSD 部署工具和端到端覆盖、NoF replica put/get 路由、L2->L1 promotion-on-hit 指标与节流参数，同时修复大对象、磁盘空间、BatchEvict、RemoveAll、hot cache 复用等边界问题。
3. **Transfer Engine 与 TENT 进入“多传输、多硬件、可调度”阶段**：TENT 增加本地 admission queue、local runtime queue dispatch、按请求 transport_hint、policy name binding、remote redis 依赖；TE 增加 Device API、TCP 强制机制、TCP slice size、IB 设备白名单、EFA CQ 线程上限、Ascend per-role 协议、Sunrise 支持等。
4. **异构硬件覆盖继续扩大**：MUSA、MACA、ROCm/HIP dmabuf、Ascend/NPU、Sunrise VRAM、CUDA Device API 等相关支持同时推进，说明项目正在把传输和 KV cache 存储能力从 CUDA/NVIDIA 主场景扩展到更多国产和异构加速器环境。
5. **稳定性和工程化投入很重**：大量提交集中在 RDMA/TCP/EFA/NVLink/NVMeoF 的错误路径、资源泄漏、竞态、超时、连接池、地址解析、UB、安全漏洞、CI 构建速度和 wheel 打包兼容性，反映 Mooncake 正在从功能验证走向生产化硬化。
6. **文档与入口重构提升上手体验**：README、Store 部署、SGLang/vLLM/LMCache 场景页、API reference、Rust API、EFA/NPU/EP/PG 文档和导航结构都有系统性更新，减少了“只看提交/代码才能用起来”的门槛。

## 新特性与产品能力

### 1. Store 多租户与管理面能力成型

本期 Store 的多租户方向变化最集中。相关提交补齐了从底层隔离到管理面的链路：租户元数据 map isolation、对象 RPC 中传播 tenant identity、tenant-aware async storage tasks、dynamic tenant quota master admission、tenant quota core，以及动态租户 quota 的 admin、eviction、metrics、snapshots。综合来看，Mooncake Store 正在从单一命名空间缓存服务，向可承载多业务、多团队、多集群隔离的共享 KV cache 存储服务演进。

这类变化的实际影响是：后续可以围绕租户做资源限额、回收策略、指标观察和快照管理，避免不同业务的 KV cache 数据在容量、元数据和后台任务上互相干扰。对生产部署而言，这是从“能跑”到“能管、能隔离、能计量”的关键一步。

### 2. SSD offload / NoF / 分层缓存能力增强

Store 的分层存储能力也明显增强。本期新增或完善了 standalone store service 的 SSD offload 配置、NoF SSD 部署工具和 e2e 覆盖、NoF replica 的 put/get 路由、SSD free-ratio-first allocation strategy、structured object copy-mode policy、size-class allocator fragmentation benchmark，以及 L2->L1 promotion-on-hit 的观测指标和 `max_per_heartbeat` 控制参数。

这些变化说明 Store 正在重点解决大容量 KV cache 场景：热数据尽量留在本地/内存层，冷数据可以落到 SSD/NoF；分配策略从简单可用空间判断，演进为更关注 SSD 空闲比例和碎片；同时通过 benchmark、metrics 和修复任务来约束分层缓存的实际效果。

### 3. TENT 调度和传输选择更细粒度

TENT 本期新增 local admission queue prototype、local runtime queue dispatch、per-request `transport_hint`、policy name binding、remote redis dependency，并持续减少 cold receive path 的 SegmentDesc 获取开销。这表示 TENT 不再只是一个固定传输后端，而是在向“每个请求可携带传输意图、运行时可按策略派发、可接入外部状态”的调度层演进。

对上层应用来说，这类能力有助于根据请求大小、拓扑、硬件和网络状态选择更合适的传输路径；对运维来说，local queue 和 policy binding 也为后续做限流、优先级、故障切换和多租户调度提供基础。

### 4. 硬件与平台支持继续扩展

Mooncake 本期新增或增强了多种硬件平台支持：EP 支持 MUSA platform，PG 增加 MUSA build 和 experimental MACA PG support，Transfer Engine 增加 ROCm/HIP dmabuf MR registration、Ascend direct async query 优化、co-located TE 的 per-role Ascend protocol、Sunrise classic path 与 tebench VRAM 支持，以及 CUDA Device API 集成。

这条线的意义在于 Mooncake 正在降低对单一 GPU 厂商和单一路径的依赖。对需要在国产 NPU、MUSA、MACA、ROCm 或混合硬件集群中部署 LLM serving/KV cache 的团队，这是重要信号。

### 5. API、运维接口和 Python wheel 体验改进

接口层面也有不少实用变化：Store 新增 `GetSegmentsDetail` admin HTTP API、`query_key` 返回合法 JSON、REST GET 保留空值、Python API 支持 zero-sized tensors、`_parse_segment_size` 支持 KB/MB/TB 后缀、console scripts async main 包装修复、uint64 async transfer API 返回文档化 failure sentinel。TE Python 包新增编译 flag 属性，TransferEngine 变为 movable，也让 C++/Python 集成更灵活。

这些变更虽然分散，但共同提升了集成体验：API 返回更稳定，边界输入更少崩溃，Python wheel 更易用，排障时可通过 admin API 和 package attributes 获得更多状态。

## 性能优化与可扩展性

### 1. Store 查询、驱逐和批处理路径优化

Store 本期引入 cached batch query result，减少批量查询重复计算；BatchGetReplicaList metadata lookup 按 shard 分组，降低元数据访问开销；lookup eviction contention 被削弱，降低并发驱逐时的锁竞争；local hot cache 注册到 transfer engine，避免数据路径上额外拷贝或不可传输问题。

此外，SSD free-ratio-first allocation strategy、size-class allocator fragmentation benchmark 和实际磁盘空间检查，都是围绕大规模存储池长期运行时的性能和容量利用率展开。相比单点优化，这更像是在补齐 Store 在高负载和长生命周期运行下的稳定性能基线。

### 2. Transfer Engine 冷路径、连接与网络传输优化

Transfer Engine/TENT 侧的性能优化集中在传输路径和连接管理：TENT cold receive path 减少 SegmentDesc fetch 开销；TCP transport 增加 `MC_TCP_SLICE_SIZE` 可配置；TCP transport 使用 `TCP_NODELAY` 并硬化 idle cleanup；handshake-port connect 增加 timeout；busy-spin 阶段通过重置 `last_wait_ts` 避免错误跳过。

EFA/RDMA/NVLink/Ascend 相关提交也在减少异常路径造成的重试、泄漏或错误设备选择。整体看，TE 不是单纯追求 benchmark 峰值，而是在把延迟、连接稳定性和异构环境适配一起推进。

### 3. CI 和构建效率优化

CI/build 侧包括 Ascend CI build speed 优化、NPU wheel 迁移到云 runner 并扩展 ARM/x86 matrix、sccache stats 容错、默认编译 flag 去掉 debug symbols、pre-release CI、Python 3.14t build 修复、Python 3.9 NPU wheel 支持、torch 2.12.1 EP/PG build matrix 等。

这些优化不会直接改变运行时 API，但会显著影响项目迭代速度和发布可靠性，特别是多平台 wheel、NPU/Ascend/MUSA/MACA 这类构建矩阵扩张后的维护成本。

## 稳定性、Bugfix 与生产化硬化

本期有大量修复集中在生产环境常见问题：

- **网络/传输错误路径**：RDMA transport 的 data race、double-close、uninit、off-by-one；TCP connection pool SIGSEGV；TCP remote address validation；EFA same-process GPU loopback；EFA GPU build caps；MR key 64-bit；NVMeoF loop/leak；MNNVL disconnect leak；HipTransport 恢复 caller device。
- **资源泄漏和异常路径**：CUDA event leak、freeaddrinfo leak、null deref、OOB read、failed io_uring sub-batch cleanup、TransferEngine error path resource leaks、PG running_ data race。
- **Store 一致性与容量边界**：BatchOffload 超 4GiB 对象整数溢出、BatchEvict 在 SSD offload 下过度驱逐、RemoveAll 未删 SSD offload 文件、cache total metrics accounting、hot cache stale reuse、larger object after block reuse、HA snapshot restore metadata version、rpc_meta republish idempotency。
- **配置和输入解析**：IPv6 endpoint parsing、ConfigDict size parsing、environment parsing、`.yml` config、signed char `tolower/toupper` UB、EFA invalid input guard、EulerOS dependencies installer。
- **服务生命周期**：mooncake_master 支持 SIGINT/SIGTERM graceful shutdown，RPC timeout，etcd watch 从 polling 改为 event-driven，并增加 leader hang 后 client recovery。

这些修复反映出 Mooncake 近期在压测、CI、多硬件部署和用户反馈中暴露了很多边界条件；项目团队正在系统性把这些问题收敛到代码和测试中。

## 易用性与文档

文档方面变化非常密集，主要不是补几句 README，而是重构信息架构：README 的 Store/SGLang/vLLM integration、hardware/build、Showcase/Components 等内容被多轮精简；文档导航和 build/design docs 重构；Store deployment guide 改为 P2P-first quick start，并补充三种 client deployment 方法；API reference 按 Python/C++/HTTP 重组，并补齐 Rust API；SGLang、vLLM、LMCache、EFA、NPU、EP、PG 相关文档都得到专项更新。

对使用者的影响是：新用户能更快按场景进入，而不是从底层组件开始摸索；已有用户也更容易定位 API、部署、硬件和性能文档。考虑到 Mooncake 正在扩展多硬件、多传输、多部署形态，这类文档重构非常必要。

## 后续关注建议

1. **重点验证 Store 多租户与 quota 的真实隔离效果**：建议覆盖跨租户高并发写入、驱逐、snapshot/restore、metrics 归属、quota 变更实时性和异常恢复。
2. **验证 SSD/NoF 分层存储在长稳压测下的容量利用率和碎片情况**：本期引入 allocator benchmark 和 free-ratio 策略，后续应关注是否改善尾延迟、写放大和空间碎片。
3. **对 TENT transport_hint / policy binding 建立推荐策略**：功能已经开始具备，但真正落地需要默认策略、示例和观测指标，否则用户很难知道何时选择 TCP/RDMA/NVLink/EFA/Ascend 路径。
4. **异构硬件支持需要持续端到端验证**：MUSA、MACA、ROCm、Ascend、Sunrise 相关提交较多，建议建立分平台 smoke/e2e 清单，避免单平台修复引入跨平台回归。
5. **关注文档与 API 实现是否同步**：本期文档重构很多，同时 API 也在快速变化，建议对 README quick start、Store deployment、Python/C++/HTTP/Rust API reference 做周期性验证。
